Overview#

83Configured checks

Text, image, nomenclature, size, legal, spell, and marketplace-specific rules

2Primary marketplaces

3Vision clients

Gemini, OpenAI-compatible APIs, and Ollama are supported

Order-based Mongo mode and direct payload mode

Rubick CAST is a product-catalog quality analysis service. It receives SKU data, chooses the marketplace rules, runs many checks, and sends structured pass/fail results back to a webhook. Think of CAST as a pipeline with four simple stages: receive product data, normalize it, run configured checks, and return a checklist with reasons.

i
Naming used in this document

The service is referred to as Rubick CAST or CAST. File paths are described using <repo-root> so setup commands remain clear even if the checked-out folder name changes.

Setup and local run commands#

Use Python 3.9 for parity with the Lambda container. Java 11 is needed when LanguageTool spell checks are enabled. Docker is needed for the production-like local run.

Development run

This runs the local HTTP wrapper. It exposes / and /health, then forwards POST requests into the Lambda handler. It also mocks self-invocation so dispatcher mode can call worker mode locally.

cd <repo-root>
python3.9 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install -r requirements.txt
pip install boto3

export OS_MONGO_URI="mongodb+srv://<user>:<password>@<cluster>/<db>"
export OS_DB_NAME="omsProd"
export PORT=8092

python local_server.py
curl http://localhost:8092/health

curl -X POST http://localhost:8092/   -H "Content-Type: application/json"   -d @payload.json

Production-like local run

This builds the Lambda container image and uses the AWS Lambda Runtime Interface Emulator behavior exposed by the base image.

cd <repo-root>
docker build --platform linux/amd64 --provenance=false -t rubick-cast:local .

docker run --rm -p 9000:8080   -e OS_MONGO_URI="mongodb+srv://<user>:<password>@<cluster>/<db>"   -e OS_DB_NAME="omsProd"   rubick-cast:local
curl -X POST "http://localhost:9000/2015-03-31/functions/function/invocations"   -H "Content-Type: application/json"   -d @payload.json
!
Local dependency note

local_server.py imports boto3, but the dependency list does not include it. Install it locally or add it to the dependency list before using the local server.

!
Secret handling note

The configuration contains provider settings. For production, secrets should be supplied through environment variables or a secret manager rather than being committed in configuration files.

Optional local vision client

Use this only when the active vision client is switched to Ollama in configuration.

# Optional local vision path, only when config active_llm_client is set to ollama
ollama serve
ollama pull qwen2.5vl:32b

Request and response contracts#

CAST accepts a request that contains root metadata plus either an orderId or direct product data. The output is not a single score; it is a set of result columns and reason columns that can be attached back to the original task or SKU.

Core request fields

formulaIdFormula/workflow identifierCopied into webhook response
formulaNameDisplay name of the formulaCopied into root metadata

operatorWorkflow operator nameCopied into root metadata
operandsIncludes marketplace selection when suppliedPreferred source for organization-specific rule filtering
webhookURLCallback URLReceives the final response from each worker
orderIdMongo order identifierTurns on Mongo batch mode
inputColumnsDirect SKU columnsUsed when no order id is supplied

Dispatcher, worker, and batch flows#

The Lambda handler has two modes. Dispatcher mode splits work into worker payloads. Worker mode processes one SKU and posts a result. This design lets an order with many tasks fan out into many independent worker invocations.

Order/Mongo mode data path

FromData movedTo
CallerSends request with `orderId`, `orgId`, `formulaId`, `operands`, and `webhookURL`Lambda handler
DispatcherReads Mongo tasks matching the orderMongoDB `tasks` collection
DispatcherLooks up stage data for every taskMongoDB `taskstages` collection
DispatcherBuilds one worker payload per task/SKUSame Lambda function, async invocation
WorkerRuns checks and builds output columnsOutput handler
Output handlerPosts result payloadWebhook URL supplied in request

Direct payload mode data path

FromData movedTo
CallerSends request with `inputColumns` and webhook metadataLambda handler
DispatcherBuilds CAST documents directly from supplied valuesWorker payloads
WorkerNormalizes source columns to CAST fieldsAnalyzer registry inside pipeline
AnalyzersRun all active rules for the marketplaceAnalysis results dictionary
Output handlerAdds result columns, reason columns, and failure summaryWebhook URL
!
Behavior to review for direct payloads

The dispatcher path treats inputColumns as items to dispatch when no orderId is present. If callers send one SKU as a list of columns, test this path carefully and adjust request shaping or dispatcher logic as needed.

Image and vision flow#

Image checks are the most expensive part of CAST. The image analyzer downloads image URLs, creates a labeled image grid, sends it to the selected vision client, validates the JSON response, then reuses that response for many image flags.

Vision response handling

StageWhat CAST doesWhy it matters
DownloadUses retry logic and browser-like headers for image URLsReduces false failures from basic server protections
MergeBuilds a labeled grid with each image slot nameThe model can refer to the correct image position in reasons
PromptAdds product attributes, marketplace instructions, restricted keyword context, and expected JSON keysKeeps the response structured
ValidateChecks JSON shape and required keys; retries malformed or incomplete responsesAvoids breaking the worker on imperfect model output
NormalizeConverts booleans and text statuses into Passed/Failed valuesMakes all check outputs consistent
CacheStores image analysis in the per-SKU processing contextAvoids sending the same grid for every image flag
!
Vision fallback behavior

When the model omits a configured key after retries, CAST currently fills that missing key as Passed with a server-side reason. This prevents a worker crash but can hide model-response quality issues, so monitor retry and missing-key logs.

Troubleshooting and implementation notes#

Common local issues

  • Local server import error: install boto3.
  • Mongo order mode returns no work: confirm OS_MONGO_URI, OS_DB_NAME, and that the order id is a valid Mongo ObjectId.
  • Spell checks fail to start: install Java 11 and enable spell checks only when needed.
  • Vision checks time out: test image URLs from the runtime network and verify the selected LLM client config.
  • Webhook receives no response: confirm workers are actually invoked and the webhook URL is reachable from the Lambda/container network.

Implementation notes to review

  • Some legal rules are configured but not implemented in the current legal analyzer.
  • Additional material/fabric and restricted-keyword checks should be reviewed against the current worker call shape.
  • The local server patches Lambda self-invocation for development only.
  • Vision fallback can mark missing model keys as Passed after retries; monitor logs.
  • Move secrets out of static configuration before production hardening.
